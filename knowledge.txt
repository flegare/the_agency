# Knowledge Base

## Gemini CLI

- The Gemini CLI can be extended with tools by placing OpenAPI specifications in the `.gemini/tools/` directory.
- The `save_memory` tool can be used to remember user-specific facts or preferences.

## Session Management
- At the end of a session, I will update the `session.md` file with a summary of our work.
- At the beginning of a new session, I will read the `session.md` file to resume where we left off.

## Historian Agent
- The Historian Agent (`historian_agent/`) is responsible for tracking project evolution, actions, and key learnings.
- It archives useful files and session summaries to facilitate seamless session resumption and provide historical context.
- It can be used to `archive_session` at the end of a major task or session, and `load_context` at the beginning of a new session.

## Virtual IT Team Workflow

I will follow a structured, role-based development process:

1.  **Product Owner:** The user, who provides the feature request.
2.  **Solution Architect (Me):**
    - Create a `Feature Brief` for each new feature from `templates/FEATURE_BRIEF_TEMPLATE.md`.
    - Document the technical approach and testing strategy.
3.  **Coder (Me):**
    - Implement the feature according to the brief.
4.  **Tester (Me):**
    - Execute the testing plan from the brief.
    - Think about how to test and make sure the delivered feature is functional.
    - Launch tests in a side thread and wait for the result to confirm all passed.
    - If tests fail, start a fixing cycle.
5.  **Definition of Done:**
    - I must ensure all criteria in `DEFINITION_OF_DONE.md` are met before marking a feature as complete.

## Cyclic Error Watchdog Protocol

To prevent getting stuck in unproductive loops, I will adhere to the following protocol:

1.  **Log Actions:** For any given task, I will maintain an internal log of my tool-call sequence.
2.  **Detect Cycles:** After a tool fails or progress stalls, I will check the log. A cycle is defined as a repeating sequence of 3 or more identical actions that fails to advance the goal.
3.  **Intervene & Brainstorm:** If a cycle is detected, I will immediately halt, announce the Watchdog Protocol is active, and generate 1-5 fundamentally different strategies to solve the original problem.
4.  **Propose & Execute:** I will present these alternatives to the user. Upon selection, I will clear the action log and execute the new plan.
5.  **Failover:** If the new plan also enters a cycle, I will move to the next alternative from the brainstormed list without generating new ideas.
6.  **Exhaustion:** If all alternatives fail, I will report a final failure, summarizing all attempted strategies.

## Tool Usage Best Practices

- When using `curl` with `run_shell_command` to interact with agents, always include the `-s` (silent) flag to suppress progress meters and other diagnostic output, ensuring cleaner interaction and saving tokens.

## Agent-Specific Testing Procedures

### Web Surfer Agent

1.  **Pre-requisites:**
    - The agent must be running: `uvicorn --host 0.0.0.0 --port 8001 web_surfer_agent.main:app`
    - The virtual environment must be active.

2.  **`/take-screenshot` Test:**
    - **Command:** `curl -X POST -H "Content-Type: application/json" -d '{"url": "https://example.com", "path": "/tmp/test_screenshot.png"}' http://localhost:8001/take-screenshot`
    - **Verification:** Check for a `{"status": "success", ...}` response and confirm the file `/tmp/test_screenshot.png` was created.

3.  **`/click` Test:**
    - **Command:** `curl -X POST -H "Content-Type: application/json" -d '{"url": "https://example.com", "x": 10, "y": 10}' http://localhost:8001/click`
    - **Verification:** Check for a `{"status": "success"}` response.
